Lesson 01 · From Zero to Live

GitHub 入門

搞懂 Git 的運作邏輯，用 CLI + GitHub 把你的作品免費發布上線。

Git 不是 GitHub。
一個是工具，一個是平台。

核心比喻

Git ＝ 你專案的時光機

一套「版本控制系統」：每一次修改都被記錄、比對、回溯。改壞了，一行指令回到昨天。

為什麼需要它

沒有 Git 混亂

  • 報告_最終版.docx
  • 報告_最終版_真的最終.docx
  • 改壞了只能 Ctrl+Z 到天荒地老

有 Git 秩序

  • 每次修改都有清楚紀錄
  • 隨時回到任何歷史版本
  • 多人改同一份不互相覆蓋

先建立的觀念

三個關鍵

  • Git ≠ GitHubGit 裝在你電腦上；GitHub 是放專案的雲端平台
  • 分散式每個人電腦都有完整歷史，斷網也能 commit
  • 追蹤的是「變更」記錄「跟上次差在哪」，不是每次存一整份檔案

GitHub 雲端

commit＝專案在某個時間點的快照（含說明、作者、時間），靠它隨時回到那個版本。

一次提交的生命週期

從修改到上雲

  • 改了 index.html — 變更在工作目錄
  • git add — 放進暫存區（告訴 Git：這個我要交）
  • git commit — 在儲存庫建立永久快照
  • git push — 上傳到遠端 GitHub

不弄壞主線的開發方式

分支 Branch

  • 在不影響主線（main）下開發新功能
  • 開新分支做實驗，搞砸直接刪，main 毫髮無傷
  • 功能完成後用 merge 合併回 main

Station 02

第二站
CLI 要裝什麼

透過終端 / AI 操作的基礎

三大工具

  • Git — 必裝版本控制核心。winget install Git.Git／brew install git
  • GitHub CLI（gh） — 強烈建議讓 CLI / AI 直接建 repo、開 PR、管 issue
  • Node.js — 視專案前端 build / npm 套件才需要

不想打指令裝？三者都有安裝檔可雙擊（Windows .msi/.exe、Mac .pkg）；gh 的在 github.com/cli/cli/releases。裝完後 gh 仍是在終端機使用。

AI 協作的關鍵一塊

為什麼是 gh CLI

  • AI 透過終端就能直接操作 GitHub，不用每次跳瀏覽器
  • 建 repo、開 PR、管 issue、查 Actions 一條龍
  • 登入一次：gh auth login → gh auth setup-git

Windows 提醒：裝 Git 會一併裝 Git Bash，多數教學指令預設用 Bash 語法，用它跑最順。

Station 03

第三站
核心使用流程

第一次使用 · 只做一次

設定身分

git config --global user.name "你的名字" git config --global user.email "你的信箱@example.com" # Windows 中文檔名不亂碼（建議） git config --global core.quotepath false

前提：已裝好 Git（見第二站）。偏好點按操作？本站最後一頁有圖形介面替代。

記這四個就夠

日常循環

git status # 看現在改了什麼 git add <檔案> # 加入暫存（git add . 加全部） git commit -m "說明" # 建立一個版本快照 git push # 上傳到 GitHub

同步別人的更新：git pull

Conventional Commits

訊息怎麼寫

  • feat: 新功能feat: 加入登入頁面
  • fix: 修 bugfix: 修正手機版排版錯位
  • docs: 文件　/　refactor: 重構　/　chore: 雜務避免 update、修改、wip 這種沒資訊量的訊息

不想碰終端機？

圖形介面替代

用指令 CLI

  • 需先裝 Git，在終端機輸入
  • add / commit / push 全手打
  • AI 在終端幫你跑的就是這條

用點的 GUI

  • GitHub Desktop — 視覺化 commit / push
  • VS Code 側邊欄原始檔控制
  • GitHub 網頁直接編輯、commit

底層都是同一套 Git；本課走 CLI，因為那正是 AI 協作的路徑。

Station 04

第四站
為什麼選 GitHub

阻力最小的選擇

六大理由

  • 最大開發者社群 — 範例、套件、協作者都最好找
  • 生態系完整 — Actions / Pages / Packages / Codespaces 一條龍
  • GitHub Pages 免費架站，自帶 HTTPS
  • gh CLI + API 成熟，AI 與自動化能完整操作
  • 業界事實標準 — 履歷、面試的共通語言
  • AI 工具深度整合 — Copilot 與各家 Agent 第一方支援

名詞解 ① · 把重複雜事自動化

CI / CD

CI 持續整合 自動檢查

每次 push，系統自動跑測試、檢查能不能 build。改壞馬上抓出來。

比喻：食材一進廚房就自動檢查有沒有壞掉。

CD 持續部署 自動上線

通過檢查後自動部署上線，不用人工上傳。

比喻：檢查通過就自動出餐到客人桌上。在 GitHub 靠 Actions 實現。

名詞解 ②

Packages / Codespaces

Packages

放你自己做的程式套件的私人倉庫，給多個專案重複安裝使用。

比喻：自研醬料裝瓶，放進公司專屬調味料櫃，誰要誰拿。

Codespaces

跑在雲端的完整開發環境，瀏覽器打開就有裝好工具的 VS Code。

比喻：租一間已備好設備的雲端廚房，開門就開工。

Station 05

第五站
GitHub Pages 實戰

純靜態 HTML · 免 build · 免 Node.js

三步驟發布

# 1. 準備好 index.html，在專案資料夾內： git init git add . git commit -m "init: 第一個靜態頁面" git branch -M main # 2. 用 gh 一行建 repo 並推上去： gh repo create 你的專案名 --public --source=. --push

網址：https://你的帳號.github.io/你的專案名/

推上去後 · 開啟託管

啟用 Pages

  • repo 頁面 → Settings → 左側 Pages
  • Source 選 Deploy from a branch
  • Branch 選 main、資料夾 / (root) → Save
  • 等 1～2 分鐘，頁面上方出現你的網址

之後改檔案只要 add → commit → push，網站自動更新。

進階 · React / Vite 等框架

自動部署

// vite.config.js — 因為網站在子路徑底下 export default { base: '/你的專案名/' } // 前後都有斜線 # 設定 .github/workflows/deploy.yml 後，push 到 main # GitHub Actions 自動 build + 部署。即時看進度： gh run watch

到 Settings → Pages → Source 選 GitHub Actions 即啟用。

帶走什麼

你現在能做的

  • 用 add → commit → push 管理任何專案的版本
  • 用 gh CLI 讓 AI 在終端幫你操作 GitHub
  • 把一個 index.html 免費發布成公開網站
